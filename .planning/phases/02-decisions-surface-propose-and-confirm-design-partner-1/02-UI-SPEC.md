## 1. Overview

### 1.1 The Surface

`/decisions` is **the owner's daily landing**. Every owner-role login redirects here. The page is a **single-column proposal inbox**: the agent (Darb) drafts actions; the owner approves, edits, or dismisses with one click. No KPI strip, no leaderboard, no chart. If the owner wants a number, they tap ⌘K and ask.

The surface is **the wedge**. Phase 2's exit criterion is design partner #1 saying *"I check Darb before WhatsApp every morning."* The whole UI is engineered for that ritual: fast load, low cognitive load, no scroll-fatigue, no buried buttons.

### 1.2 Personas Touched in Phase 2

| Persona | Surface | Phase 2 Behavior |
|---------|---------|------------------|
| Owner / GM | `/decisions` (landing) + `/decisions/audit` | Reviews 5–25 proposals each morning; approves/dismisses. |
| Admin / Founder (super-admin role) | `/admin/onboarding` + `/admin/billing` | Provisions design partner #1 tenant; runs the 30-day report; charges KD 100/month. |
| Dispatcher | Sidebar still shows the legacy `Triage` link in Phase 2; `/floor` ships in Phase 7. No regression. | — |
| Accountant | No new surface in Phase 2; `/finance/cash` ships in Phase 8. | — |

### 1.3 User Flow — Owner's Morning Ritual

```
06:00  Agent's monitoring loop has run all night (every minute since 22:00)
06:30  Owner opens https://app.darb.kw → JWT redirects role=ADMIN/OWNER → /decisions
06:30  Page renders 12 cards in priority order (mix of penalty / cash reminder / warn / promote / suspend)
06:31  Owner reads card 1: "Mohamed (driver_xy12) — 3 late clock-ins this week. Suggest written warning."
06:31  Owner clicks Approve → optimistic UI flips card to "Approved (just now)" with undo for 5s
06:31  AgentAction row written: proposer=Darb, approver=user_42, toolName=draftCourierMessage, outcome=success
06:32  Card 2: "Cash reminder: Ali (driver_z73) owes KD 28.500 from Keeta settlement"
06:32  Owner clicks Edit → drawer opens with editable WhatsApp body + amount → owner trims to KD 25 → Save & Approve
06:33  Card 3: "Suspend driver_aa11 — 5 cancellations today, GPS-stale 4 times this week"
06:33  Owner reads, decides this driver had a phone repair — clicks Dismiss with reason "phone repair, told dispatcher"
06:33  AgentMemory row written: key=`dismissed:suspend:driver_aa11`, value={reason}, confidence=0.9
06:34  Owner finishes inbox, closes laptop. Total time: 3m 30s.
```

Designed-for-time-on-task: 5–10 cards in 2–3 minutes, 25 cards in 6–8 minutes.

#### 3.1.2 Decision Card — Anatomy (CON-decisions-card-shape)

| Region | Content | Specification |
|--------|---------|---------------|

| **Confidence** (top-left, after tag) | `85% confidence` | `text-[11px] text-secondary`. Hidden if confidence < 0 (defensive). |
| **Shortcut hint** (top-right) | `Cmd+{N}` where N = card index 1-9 | `text-[11px] font-mono text-sand-500`. Only visible when card is the focused card (kbd-nav). |
| **Headline** | Driver-name + 1-sentence summary | `text-[15px] font-semibold text-foreground leading-snug`. Driver name is a `<Link>` to `/drivers/[id]` (Phase 3 — for now, link is a placeholder showing an alert "Driver File ships in Phase 3"). Headline truncates at 90 chars with ellipsis; full text in a tooltip. |
| **Reasoning** | 2 lines max of agent's plain-English explanation | `text-sm text-sand-700 leading-relaxed`. Truncate at line 2 with `line-clamp-2`. Click to expand inline (no navigation). |
| **Evidence disclosure** | Collapsible "Show evidence (N)" toggle revealing structured data: links to specific shifts / cash records / violations | `text-xs font-medium text-primary` toggle. When expanded, renders `<EvidenceList />`: each item is a row with icon + label + linked entity ID. |
| **Audit-row preview disclosure** | Collapsible "Audit-row preview" toggle revealing the JSON-shaped row that will be written to `AgentAction` on approve | `text-xs font-medium text-primary` toggle. #### 3.1.5 Card State Transitions (in-page, no reload)

```
┌──────────┐   approve     ┌──────────┐
│ pending  │ ────────────► │ approved │ (5s undo window with toast)
└──────────┘               └──────────┘
     │                          │
     │ dismiss                  │ undo (within 5s)
     ▼                          ▼
┌──────────┐               ┌──────────┐
│ dismissed│               │ pending  │
└──────────┘               └──────────┘
     │
     │ (no undo — dismissal writes to AgentMemory)
     ▼
   archived (faded card slides out after 1.5s)
```

**Visual on transition:**
- `approved`: card background flips from white to `bg-primary/5`; tag chip animates to a checkmark icon `bg-primary text-white`; action footer is replaced with `Approved {N}s ago by you · Undo`.
- `dismissed`: card opacity drops to 0.5; tag chip becomes `bg-sand-200`; action footer is replaced with `Dismissed: "{reason}" · {timestamp}`.
- After 1.5s in approved state without undo, card slides out (transform translateX + opacity 0 over 250ms ease-sierra-out) and remaining cards reflow upward. After 1.5s in dismissed state, card collapses height to 0 and disappears.

#### 3.1.7 Loading State

- **Initial load**: render 5 skeleton cards using existing `<Skeleton />` component. Each skeleton matches card height (~180px) with shimmer animation.
- **Refresh** (after approve/dismiss): no full reload; only the affected card animates out.
- **Background poll** (every 30s): silent. New cards fade in at the top with a 1px primary border that fades after 1s, drawing attention without being jarring.
- **Network failure** during initial load: render `<ErrorState />` (existing component) with `Retry` button. Body: "Couldn't load proposals. Check your connection or try again."

#### 3.1.8 Error State Within a Card

If `POST /api/decisions/:id/approve` returns 4xx/5xx:
- Card stays in `pending` state.
- Tag changes to a one-shot amber `bg-amber-50 text-amber-700` "Couldn't approve — retry" pill that **replaces the confidence display for 8 seconds**, then restores.
- A toast (existing `<Toast />` provider) fires: "Couldn't approve {action}: {server message}". Variant: `error`.
- The Approve button remains active for retry.

## 5. Interaction Patterns

### 5.1 Optimistic Updates

| Surface | Behavior |
|---------|----------|
| Approve | Card flips to `approved` immediately. Server call fires; on 200, undo button stays for 5s then card slides out. On 4xx/5xx, card reverts to `pending` + amber error chip + toast. |
| Dismiss | Card flips to `dismissed` immediately. Server call fires async. On error, card reverts + toast. |
| Edit save | Drawer closes, card stays in `pending` (with edit indicator), then approve fires per Approve flow. |
| Override toggle (billing) | Toggle flips immediately. On error, toggle reverts + toast. |

### 5.2 Loading States Per Surface

| Surface | Loading representation |
|---------|------------------------|
| `/decisions` initial load | 5 `<Skeleton />` cards (180px tall each) with shimmer |
| `/decisions` background poll | No visible state; new cards fade-in at top |
| `/decisions/audit` initial | DataTable's built-in `loading=true` (spinner row) |
| `/admin/onboarding` step transitions | Step button shows spinner; disabled until response |
| `/admin/onboarding/backwash` job | Per-platform progress bars + "Pulling {platform}…" text |
| `/admin/billing` initial | DataTable spinner |

### 5.3 Real-Time Updates (Phase 2 scope)

**Decision:** Phase 2 uses **30-second polling** (`GET /api/decisions?status=pending&limit=25`), not SSE/WebSocket. WebSocket ships in Phase 4 alongside the chat surface; the Decisions page upgrades to live SSE when Phase 4 lands.

Rationale: Phase 2 owner usage is checked-in-the-morning, not always-on; 30s polling is sufficient and keeps Phase 2 scope tight.

### 5.4 Undo Window

Approve action shows a sticky toast for 5 seconds:

```
┌──────────────────────────────────────────┐
│ ✓ Approved: Warn Mohamed       [Undo]    │
└──────────────────────────────────────────┘
```

If clicked within 5s:
- Calls `POST /api/decisions/:id/undo`
- Backend marks `AgentAction.outcome = "rolled_back"`, `rolledBackById = userId`, `rollbackReason = "user-undo"`.
- Toast disappears, card flips back to `pending`.

### 5.5 Conflicting Proposals (edge case)

If two proposals target the same driver with conflicting actions (e.g., a `Warn` and a `Suspend`):
- Both render as separate cards.
- The card UI shows a `RelatedConflict` chip: `"⚠ 1 related proposal — review together"` linking to the other card.
- Approving one does NOT auto-dismiss the other; user must explicitly act on each. (Rationale: agent might be right to escalate; let owner decide.)

### 5.6 Dismiss-to-Trained-Against Semantics

Per **REQ-decisions-proposal-inbox**:

When user dismisses, write to `AgentMemory`:
```json
{
  "tenantId": "...",
  "key": "dismissed:{toolName}:{subjectType}:{subjectId}",
  "value": {
    "reason": "Phone repair — known absence",
    "dismissedBy": "user_42",
    "dismissedAt": "2026-05-09T06:33:08Z",
    "originalProposal": { ... }
  },
  "confidence": 0.95
}
```

The continuous-monitoring loop reads this on next scan and **suppresses identical proposals for 7 days** (configurable later in Phase 12 standing rules).

## 6. Accessibility (WCAG 2.1 AA)

| Dimension | Requirement | Implementation |
|-----------|-------------|----------------|
| **Keyboard nav** | Every interaction reachable without mouse | All buttons have visible focus rings (`focus:ring-2 focus:ring-primary/40`); arrow keys move focus between cards; tab order: filter chips → sort dropdown → cards → action buttons within card |
| **Focus indicator** | 3:1 contrast ratio against adjacent colors | Use `ring-2 ring-primary/40` (forest at ~38% alpha) — meets AA on white card |
| **ARIA labels** | All icons + non-text controls labeled | Each card is `role="article"` with `aria-labelledby={card-N-headline}`; tag pills have `aria-label="{tag} proposal"`; Approve/Edit/Dismiss buttons have `aria-label="{action} {tag} for {driverName}"` |
| **ARIA live regions** | Card state changes announced | `aria-live="polite"` on `<DecisionsList />` container; on approve/dismiss, append a hidden `<span>{action} for {driverName} {state}</span>` for screen reader |
| **Color contrast** | All text ≥ 4.5:1 against background | Body text `#302E2D` on `#FFFFFF` = 14.4:1 ✓; secondary `#716F6C` on `#FFFFFF` = 5.5:1 ✓; tag chips checked individually below |
| **Tag chip contrast** | All chip text ≥ 4.5:1 | Penalty `text-red-700 (#B91C1C)` on `bg-red-50 (#FEF2F2)` = 5.0:1 ✓; Cash `text-sky-700` on `bg-sky-50` = 4.7:1 ✓; Warn `text-amber-700` on `bg-amber-50` = 4.6:1 ✓; Promote `text-primary (#006838)` on `bg-primary/10` = 4.7:1 ✓ |
| **Modal focus trap** | EditDrawer + ConfirmModal trap focus | Existing `<ConfirmModal />` already implements; `<SlidePanel />` adds a focus trap in Phase 2 (TODO for executor) |
| **Reduced motion** | Honor `prefers-reduced-motion` | All slide-in / slide-out animations gated behind `@media (prefers-reduced-motion: no-preference)`; with reduced motion, transitions are instant |
| **Color-independent state** | Approved / dismissed not signaled by color alone | Approved card shows a `Check` icon + "Approved" text; dismissed shows `XCircle` icon + "Dismissed" text |
| **Form errors** | Errors announced and linked to field | Onboarding form fields use `aria-describedby={field-error-id}`; errors in red text below input |
| **Skip link** | Skip nav at top of page | Reuse existing skip-link if present; add `<a href="#main">Skip to main content</a>` if missing |
| **Heading hierarchy** | One h1 per page; no skipped levels | `/decisions` h1 = "Decisions"; cards use h3 for headline |
| **Touch targets** | ≥ 44×44 px | Action buttons are 40px tall × ~96px wide ✓; tag pills are 22px tall — labeled, not interactive (clicks bubble to card focus); filter chips are 28px tall × wider — borderline; **bump to 32px tall in implementation for AA compliance** |

## 9. Edge Cases

| Case | Behavior |
|------|----------|
| **Empty inbox** | `<DecisionsEmptyState />` rendered (see §3.1.6) |
| **Filter applied + 0 results** | Empty state with copy: `"No {filter} cards. Try All."` + clear-filter button |
| **Network failure on initial load** | `<ErrorState />` with Retry button; preserve filter state in URL |
| **Network failure during approve** | Card stays pending; amber error chip; toast; Approve button retry-able |
| **Network failure during dismiss** | Card flips back to pending; toast; user can retry |
| **Agent disabled** (super-admin globally disables agent) | Page shows banner: `"⚠ Agent monitoring paused by admin. New proposals won't appear until resumed."` Existing pending cards remain operable. |
| **Tenant has no couriers** | Empty state body changes to `"You haven't imported any couriers yet. {Link: Onboarding}"` (visible only to ADMIN) |
| **Conflicting proposals** | Both cards render; `RelatedConflict` chip on each linking to the other |
| **Stale card** (poll returned a card the user already approved on another device) | Server returns 409 on approve; UI gracefully merges: card flips to "Approved by {otherUser} {time}" + auto-dismisses |
| **Card for archived/deleted driver** | If `driver` is null on render, skip the card and emit a backend warning log (don't surface to user) |
| **Race: approve + dismiss on same card** | Backend uses optimistic-lock on AgentAction-pending row; second action returns 409; UI shows: `"This proposal was already {action} a moment ago."` |
| **Edit drawer with invalid data** | Save button disabled; field-level errors shown |
| **Onboarding backwash job fails** | Step 4 shows error state with retry; failed scrapers listed individually |
| **Billing override exceeds floor** | Override > KD 200 default flagged with warning toast: `"Override raises bill above standard floor. Confirm?"` |
| **Page accessed by wrong role** (e.g., VIEWER tries to approve) | Server returns 403; UI shows toast + reverts; admin link in sidebar hidden for non-super-admins |
| **Long reasoning text (> 200 chars)** | `line-clamp-2` truncates; click-to-expand works inline |
| **Driver name very long (> 60 chars)** | Truncate with ellipsis; full name in tooltip |
| **Approval triggers an irreversible side-effect on tool not in Phase 2 scope** | Tool not callable in Phase 2. Card UI disables Approve with tooltip `"Action tool ships in Phase 8 — your approval is recorded for training only."`; dismiss/edit still work. |
