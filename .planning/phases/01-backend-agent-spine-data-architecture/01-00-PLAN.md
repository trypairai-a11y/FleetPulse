## Threat Classes Surfaced from Phase 1 Security Requirement

- **Tenant-isolation bypass**: Addressed by Wave 0 (this plan): the lint rule + tenantIsolation.test.ts. Wave 3 + Wave 4 turn these tests green.
- **Tool-call injection (malformed input)**: Addressed by strict.test.ts ensuring every tool has strict schemas; Wave 3 implementations carry Zod validators as a second line.
- **Audit-log tampering**: Out of Wave 0 scope (Wave 2 ships the `AgentAction` schema and writer; the WORM constraint is documented in Wave 4's plan but enforced by `prismaExtensions.ts` runtime guard, NOT a static check in Wave 0).
- **PII in agent memory**: Out of Wave 0 scope (Wave 2 ships the `AgentMemory` schema; retention policy is a Phase 11 deliverable per RESEARCH.md "Out of Scope").
- **Prisma client extension bypass**: Wave 0 lint rule catches `prisma.<scopedModel>.findMany` calls but does NOT catch `prismaUntyped.driver.findMany` (a hypothetical raw client). The ONLY way to obtain a raw Prisma client is via `backend/src/config/prisma.ts` which already wires `prismaExtensions`; this is a known assumption (RESEARCH.md A1).
