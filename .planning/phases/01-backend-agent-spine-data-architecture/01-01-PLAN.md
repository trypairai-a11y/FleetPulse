<objective>
Land the foundational data plumbing AND consolidate the agent module under its new home.

This wave executes two orthogonal-but-coupled streams:

**Stream A — Schema additions (REQ-data-agent-action, REQ-data-agent-memory, REQ-data-pinned-view, REQ-data-performance-snapshot, REQ-data-metric-event):**
Add 5 new Prisma models to `backend/prisma/schema.prisma` with full tenant scoping, indices, and relations. Update `prismaExtensions.ts` to include them in `TENANT_SCOPED_MODELS`. **Migration is NOT run in this wave** — it lands in Wave 4 as the [BLOCKING] schema push task. Wave 1 only EDITS the schema file.

**Stream B — Module relocation (DEC-promote-agent-to-spine):**
Move `backend/src/services/agents/` to `backend/src/agent/`, rename internal files (`agentRuntime.ts` -> `runtime.ts`, `agentScheduler.ts` -> `scheduler.ts`, `toolRegistry.ts` -> `registry.ts`), keep all 4 registered agents and 3 tool files (`triage`, `reconciliation`, `narrator`) functionally identical. Leave a re-export shim at the old path so any caller we missed continues to compile (Wave 4 deletes the shim after `aiChiefOfStaffService.ts` is gone). Add a centralized `agent/config.ts` that exports `MODEL = "claude-sonnet-4-6"` and `MAX_ITERATIONS_DEFAULT = 8`.

Purpose: Schema change + module rename are mechanically simple but high-touch (many imports update at once). Doing both in a single wave lets us verify the system still compiles + existing tests still pass + Wave 0 schema.test.ts goes green BEFORE Wave 2 starts writing data primitives. Stream A's migration is deferred to Wave 4 because we want all model field changes (potentially across waves) batched into a single `prisma migrate dev` call.

Output: A schema.prisma file with 5 net-new models, a fully-relocated `backend/src/agent/` directory, all callers updated to the new import path, and `npm test` continues to pass on the existing test suite.
</objective>
