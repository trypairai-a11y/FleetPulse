## 1. Overview

### 1.1 The Surface

`/drivers/[id]` is **the source of truth before any owner approves a fire / promote action**. When the owner sees a `Suspend Mohamed` proposal in `/decisions`, they click `Mohamed` and land here. When the dispatcher sees a red dot on `/floor` (Phase 7), they click the dot's name and land here. When the agent generates a leaderboard table in `/chat` (Phase 4), every driver name links here.

The page is **read-mostly with one writeable surface**: the agent's plain-English score explanation comes from a chat-style "Ask Darb why?" trigger inside the Score section. Every other surface is read-only — penalty / suspend / promote actions live in `/decisions` or `/chat`, not here.

The eight required sections render top-to-bottom on a single scroll. A scroll-spy sidebar pins to the right and tracks the current section. The page is keyboard-navigable: `g` then `1`–`8` jumps to a section.

### 1.2 Personas Touched in Phase 3

| Persona | Surface | Phase 3 Behavior |
|---------|---------|------------------|
| Owner / GM | `/drivers/[id]` after clicking a name in `/decisions` | Reads the file before approving suspend / promote / penalty proposals. Looks at score trend, violations, cash, then back to inbox. |
| Ops manager / dispatcher | `/drivers/[id]` after clicking a name in chat-generated tables (Phase 4) | Quick triage during the day. Reads Live section + Recent Orders. |
| Accountant | `/drivers/[id]` after clicking a name in `/finance/cash` (Phase 8) | Reads Cash section (settlements + outstanding) and the agent's notes for context. |
| Founder (super-admin) | Used during white-glove onboarding to walk a prospect through one of their own driver files | Demo surface — same page, no special mode. |

### 1.3 User Flow — Owner's Pre-Approval Ritual

```
06:35  Owner is in /decisions. Card #4: "Suspend Ahmed (driver_bb22) — 5 cancellations
       today, GPS-stale 4 times this week"
06:35  Owner clicks "Ahmed" link in card → routes to /drivers/bb22
06:35  Page renders. Header shows photo, status pill (offline 28m), today's count
       (3 / 5 expected). Score: 47 / 100 (red trend arrow ↓ -12 over 30d).
06:36  Owner scrolls to 90-Day Trend. Sees the dip started 18 days ago. Clicks
       "Ask Darb why?" inline in the Score section.
06:36  Inline drawer expands with agent's plain-English explanation:
       "Ahmed's score dropped because (1) attendance dipped to 71% last 30d
       (was 92%), (2) 4 GPS-stale flags this week — phone may be off,
       (3) 3 late pickups in the last 14d. The drop started after the
       18 Apr off-day request was rejected — emotional cause is plausible."
06:37  Owner scrolls to Violations. Sees 2 open + 5 resolved. The 2 open are
       both "GPS_NOT_UPLOADING" — confirms agent's hypothesis.
06:37  Owner scrolls to Cash. Outstanding: KD 12.250 (overdue 4 days from Keeta).
06:38  Owner scrolls to Agent's Notes. Sees memory entry from 19 Apr:
       "Ahmed asked supervisor for 2 days off — denied due to short staffing.
       Owner may want to address."
06:38  Owner clicks ⌘← (or back button) → returns to /decisions
06:39  Owner clicks "Edit" on the suspend card → drawer opens; owner changes
       suspend duration from 7 days to "Warn instead, schedule 1:1 with supervisor"
06:39  Owner saves & approves modified action. Total time on driver file: ~3min.
```

Designed-for-time-on-task: 60–180 seconds per driver. Most owners spend ≤90s, scrolling, scanning, deciding.

### 2.2 The Global `<DriverLink>` Component

**Single source of truth for "make every driver name clickable."** Replaces every ad-hoc `<Link href={...} />` and every plain `<span>{driverName}</span>` across the app.

```tsx
// frontend/src/components/shared/DriverLink.tsx (NEW in Phase 3)
import Link from "next/link";
import { cn } from "@/lib/cn";

interface DriverLinkProps {
  driverId: string;
  driverName: string;
  platform?: "keeta" | "talabat" | "deliveroo" | "americana";
  variant?: "inline" | "card-headline" | "table-cell" | "subtle";
  showPlatformDot?: boolean;
  className?: string;
}

export default function DriverLink({
  driverId, driverName, platform, variant = "inline",
  showPlatformDot = false, className,
}: DriverLinkProps) {
  const href = platform
    ? `/drivers/${driverId}?platform=${platform}`
    : `/drivers/${driverId}`;
  const baseClass = cn(
    "inline-flex items-center gap-1.5 transition-colors duration-250 ease-sierra-out",
    variant === "inline" && "text-primary hover:underline decoration-primary/40 underline-offset-2 font-medium",
    variant === "card-headline" && "text-foreground font-semibold hover:text-primary",
    variant === "table-cell" && "text-foreground hover:text-primary",
    variant === "subtle" && "text-secondary hover:text-foreground",
    className,
  );
  return (
    <Link href={href} className={baseClass} prefetch={false}>
      {showPlatformDot && platform && (
        <span className={cn("w-1.5 h-1.5 rounded-full",
          platform === "keeta" && "bg-keeta",
          platform === "talabat" && "bg-talabat",
          platform === "deliveroo" && "bg-deliveroo",
          platform === "americana" && "bg-americana",
        )} />
      )}
      {driverName}
    </Link>
  );
}
```

### 5.2 "Ask Darb Why?" Trigger Flow

```
1. User clicks "Ask Darb why this score?"
2. Drawer expands; loading skeleton + "Darb is thinking…"
3. Frontend calls GET /api/drivers/:id/score-explanation
4. Backend:
   a. Check cache key `score_explanation:{tenantId}:{driverId}:{snapshotDate}` (1h TTL)
   b. If cached, return it
   c. Else: call agent run with prompt template + courierProfile tool result
   d. Stream back response
   e. Write AgentRunLog row (visibility: agent saw this)
   f. Return the explanation
5. Frontend renders text in drawer
6. User can click "Refresh" → invalidate cache, regenerate
7. Drawer state persisted in component memory; closing the drawer does not lose the cached text
```

The endpoint is **read-only** (does not write to AgentAction). The agent run is logged in AgentRunLog so we can see how often the explanation is requested per driver — feeds into Phase 11 mature-memory work.

### 5.3 Scroll-Spy Behavior

- IntersectionObserver watches each `<section>` (rootMargin: `-25% 0px -50% 0px`).
- When a section enters the viewport, its scroll-spy entry becomes active (`text-primary font-medium` + leading filled circle `●`); inactive entries `text-secondary`.
- Clicking a scroll-spy entry smooth-scrolls to the section (`scrollIntoView({ behavior: 'smooth', block: 'start' })`).
- Keyboard: `g` then `1`–`8` jumps to section N. `Esc` closes any expanded inline drawer.
- Reduced motion: smooth-scroll becomes instant.

### 5.4 Optimistic Updates

Driver File is read-mostly. The only optimistic update is **copy-to-clipboard** on platform IDs:
- Click → instant toast `"Copied {platformId}"` even before clipboard API resolves
- If clipboard API fails (e.g. permission denied), toast updates to `"Couldn't copy — manually select"` with retry CTA

### 5.5 Error Recovery

**Per-section retry:** If a single section's fetch fails (e.g. cash-snapshot 500), only that section renders an inline error. Other sections continue. Retry button calls only that endpoint.

**Page-level retry:** If `GET /api/drivers/:id/profile` (the bulk endpoint) fails, full-page error. Retry button reloads the page.

**Stale-while-revalidate:** Per-section endpoints use `staleTime: 0, refetchOnWindowFocus: true` for live-changing data (cash, orders, live status). Static data (trend, snapshot) uses `staleTime: 30 * 60 * 1000` (30 min).

### 5.6 Cross-Tenant Hardening

Per **CON-tenant-scope-everywhere**:
- Backend MUST 404 (not 403) on cross-tenant access to avoid leaking tenant existence.
- Frontend treats 404 and 403 identically — same `<ErrorState />` "Driver not found".
- The route guard in `app/(dashboard)/drivers/[id]/page.tsx` does NOT pre-check tenant; the API endpoint is the source of truth.

|-------------|----------|
| `?platform=keeta` | Sets default `<RecentOrdersTable />` filter to Keeta. Otherwise no effect. |
| `?notes=observations` | Pre-selects Observations tab in `<AgentNotes />`. |

## 8. API Contracts

All endpoints are tenant-scoped via existing `tenantScope` middleware.

### 8.1 Bulk Profile Endpoint

`GET /api/drivers/:id/profile?platform={platform}` — returns the entire `DriverFileData` shape in one call.

**Why bulk?** Phase 3 success criterion 3 requires the page to render <300ms. A single round-trip with parallel internal queries beats 8 chatty endpoints. Existing `courierProfile` agent read tool already aggregates 4 of the 8 sections; this endpoint extends that pattern.

**Response shape:** the `DriverFileData` interface from §4.5.

**Internal implementation:**

```ts
// backend/src/routes/drivers.ts (new handler)
router.get("/:id/profile", async (req, res) => {
  const { id } = req.params;
  const tenantId = req.tenantId; // from tenantScope middleware
  const platform = req.query.platform as Platform | undefined;

  const driver = await prisma.driver.findFirst({
    where: { tenantId, id },
    select: { /* identity columns */ }
  });
  if (!driver) return res.status(404).json({ error: "driver_not_found" });

  // Run 8 queries in parallel
  const [
    liveStatus, score, trend, shifts, orders,
    violations, cash, notes,
  ] = await Promise.all([
    getLiveStatus(tenantId, id),
    getLatestSnapshotWith7dDelta(tenantId, id),
    getNinetyDayTrend(tenantId, id),                    // PerformanceSnapshot rows
    getRecentShifts(tenantId, id, { days: 14 }),
    getRecentOrders(tenantId, id, { limit: 50, platform }),
    getViolationsBuckets(tenantId, id),
    getCashSnapshot(tenantId, id, platform),
    getDriverNotesBuckets(tenantId, id),
  ]);

  return res.json({ profile: maskCivilId(driver), liveStatus, score, trend,
    shifts, orders, violations, cash, notes });
});
```

**Latency budget:** <300ms p50, <800ms p99 with 90 days of snapshots and 50 orders. Add an integration test asserting this on a seeded driver.

### 8.3 Trend Endpoint (already covered by bulk; standalone for refresh)

`GET /api/drivers/:id/trend?days=90&metric=composite` — returns `trend` slice. Optional standalone for the future case where the chart's metric selector triggers a different metric column. Phase 3 implementation detail: keep one chart-data fetch on mount; metric selector toggles client-side from already-fetched multi-metric data (the bulk endpoint returns all 6 metrics per point).

### 8.4 Score Explanation Endpoint (NEW agent-flow)

`GET /api/drivers/:id/score-explanation`

**Returns:** `ScoreExplanation` interface (§4.5).

**Flow:**

1. Check Redis cache `score_explanation:{tenantId}:{driverId}:{snapshotDate}` (TTL 1h).
2. If cached → return.
3. Else:
   - Load latest `PerformanceSnapshot` for driver
   - Load `courierProfile` tool result (already-cached if recent)
   - Build prompt template:
     ```
     System: You are Darb, the AI ops chief. Explain why a driver received a
     specific composite score. Be concrete: cite which sub-score is dragging,
     mention the absolute number, mention any recent change in context. ≤4
     short paragraphs. Plain English. No emoji.

     User: Driver {name}, composite score {composite}/100. Sub-scores:
     attendance {a}/100, delivery {d}/100, financial {f}/100, equipment {e}/100,
     platform {p}/100. Trend: {trend} ({delta} over 7d).
     Recent context: {courierProfile JSON}.
     Why is this score {composite}? What changed? What sub-score is dragging?
     ```
   - Run via Claude Sonnet 4.6 (existing agent module integration).
   - Cache result with TTL 1h.
   - Write `AgentRunLog` row with `runType="score_explanation"`.
4. Return `{ text, generatedAt, modelName, latencyMs, cachedUntil }`.

**Endpoint is `requiresApproval: false`** — it's a read-only explanation, no action against the world.

### 8.5 Endpoint Authorization

All endpoints enforce:
- `authMiddleware` (existing): valid JWT
- `tenantScope` (existing): `req.tenantId = jwt.tenantId`; the WHERE clause must include `tenantId: req.tenantId`
- `rbac(...)` (existing): for `/profile` and all section endpoints, allow all roles (`OWNER`, `ADMIN`, `OPS_MANAGER`, `SUPERVISOR`, `ACCOUNTANT`, `VIEWER`)
- Civil ID masking happens server-side; only `ACCOUNTANT` role gets unmasked civil ID via the existing `/api/drivers/:id/full` endpoint (out of scope for Phase 3 — keep masked everywhere on `/drivers/:id`)

## 9. Edge Cases

| Case | Behavior |
|------|----------|
| Driver belongs to another tenant | API returns 404 (not 403) to avoid leaking tenant existence. Frontend renders "Driver not found" |
| Driver was just hired (0 days of snapshots) | Score gauge `—`; trend section shows "Not enough history yet" empty panel; other sections work normally |
| Driver has 8 days of snapshots (between 0 and 14) | Score gauge shows latest; trend shows fallback empty panel with hire date |
| Driver has gaps in PerformanceSnapshot (e.g. worker missed 3 days) | Trend chart bridges gaps (`connectNulls={true}`); footer shows "3 days missing data — bridged in the chart" warning |
| Driver is Americana | Cash section replaced with informational empty card; orders & violations still render normally |
| Driver is multi-platform (Keeta + Talabat) | Header shows multiple platform-id pills; orders section defaults to All; cash section shows cumulative across Keeta + Talabat (CON-cash-platform-coverage excludes Americana) |
| Driver photo URL is broken / 403 | Fallback to initials avatar; `<img onError>` swaps to fallback |
| Civil ID is null on record | Skip civil-ID line entirely; do not render placeholder |
| Driver has no platform IDs synced | Show single chip "No platform IDs synced yet" + link to /admin/onboarding |
| Driver name contains Arabic glyphs | Renders inline via `font-sans` fallback; no `dir="rtl"` switching needed for v1 |
| Driver name >60 chars | Truncate at 60 with `…`; full name in tooltip |
| Score = 0 (totally inactive) | Gauge fills to 0% (red band); sub-scores all 0; "Ask Darb why?" still works, agent explains "no data → score 0" |
| Score = 100 (perfect) | Gauge fills to 100%; trend arrow shows ↑; "Ask Darb why?" praises performance |
| Snapshot computed_at is >36 hours ago (worker stuck) | Show amber chip in score header: `Snapshot stale — last computed {N}h ago` |
| 90-day trend has all-null compositeScore (worker never ran) | Empty panel "Snapshot worker hasn't run yet — check /admin/onboarding" |
| 0 violations ever | All tabs empty; show "No violation history" |
| 0 orders ever | "No orders yet. Driver may be new or inactive." |
| Cash outstanding is negative (overpaid) | Show in green: `KD -5.000 (overpaid)`. Sub-text: "Last settlement was overpaid by 5 KD; will be reconciled in next cycle." |
| 50+ recent orders all from one platform | Filter chip count shows e.g. "Keeta 50"; All chip also shows 50; other platform chips show 0 (still rendered, click renders "No {platform} orders. Try All.") |
| Notes tab with 0 entries | Per-tab empty message (see §3.9.6) |
| Audit log with > 50 rows | 50 most recent + "View full audit log →" link to filtered audit page |
| User clicks "Ask Darb why?" while page is still loading | Disable trigger; show tooltip "Score loading…" until snapshot is fetched |
| Concurrent edit elsewhere (e.g. owner approves a suspend in another tab) | Driver File polls `liveStatus` every 30s; after suspend, `status` reflects `SUSPENDED`. AgentNotes refetch does NOT happen until tab change → owner won't see the new audit row until manual refetch. Acceptable for v1. |
| Network slow (3G simulation) | Skeleton renders <500ms; bulk endpoint times out at 10s; on timeout show error state with retry |
| Civil ID would-be visible inadvertently in any console.log or error trace | Add ESLint rule `no-civil-id-leak` checking for civil-ID-shaped strings (`/^\d{12}$/`) in non-API-handler files. Adopt pattern from Phase 1's tenant-scope ESLint rule. |
| Browser back from `/decisions` → `/drivers/[id]` → back to `/decisions` should preserve filter state | Phase 3 inherits Phase 2's URL-param-driven filter state. Browser back works natively. |
| Mobile-app (Phase 5) doesn't exist yet | Status pill source is the existing scraper / shift status — same as Phase 2 monitor agent. No regression. |
