## 1. Overview

### 1.1 The Surfaces (three, not one)

Phase 4 ships **three complementary chat surfaces** that share state through React Query + a thread-id URL param:

1. **⌘K Command Palette** (`<AskDarbPalette />` v2). Mounted globally in the dashboard shell. Opens on `Cmd+K` / `Ctrl+K` from anywhere. Frame: 760px wide overlay, `top: 12vh`, `rounded-2xl`. Use cases: short questions ("how many drivers online?"), recent-thread switching, quick-action launcher (4 preset prompts). **Does not** render mini-dashboards inline — instead, shows a 1-line answer with a "View full answer →" CTA that opens `/chat?thread=…` in the same tab. Replaces the current `AskDarbPalette.tsx` (which today only renders a chat list).
2. **`/chat` full-screen surface** (`<ChatPage />`). The primary place where mini-dashboards render. Three-column layout: thread sidebar (left) · message thread (center) · contextual right panel (collapsed by default; appears for action proposals, pinned-view previews, and "Save as briefing" forms). Replaces the existing `/copilot` page.
3. **`/decisions` "Pinned views" rail** — Phase 4 adds a collapsible header rail above the existing card stack on `/decisions`, rendering the user's pinned tiles. Each tile is the same generated-view component used inline in chat (read-only mode, plus an "Open in chat" link).

### 1.3 User Flow — Owner's "Why did revenue drop?" ritual (the worked example from PRD §5.3)

```
06:00  Morning briefing job runs (BullMQ schedule), writes 1 thread to ChatThread + 1 message
       to ChatMessage with viewType=kpi_strip + viewType=callout summarising yesterday.
06:32  Owner finishes /decisions inbox (Phase 2 ritual). Notices revenue tile down 12% in the
       Pinned views rail (rail is collapsed by default; rendered as 3 tiles).
06:32  Owner clicks the briefing thread in sidebar — opens at /chat?thread=brf_2026_05_10
       The morning brief renders as 3 stacked viewBlocks: kpi_strip, callout, action_card.
06:33  Owner types "Why did revenue drop yesterday in Hawally?"
       Assistant streams response via WS:
         (a) kpi_strip viewBlock with 3 tiles (Revenue, Courier coverage, Reject rate) — fades in
         (b) bar_chart viewBlock with Tue-vs-Wed by hour — animates
         (c) callout viewBlock (amber) naming the 2 no-show drivers — fades in
         (d) actionable footer with 3 buttons: Draft warnings · Reassign tomorrow · Pin to Home
06:34  Owner clicks "Draft warnings" → inline ChatProposalCard renders below the assistant msg
       showing 2 driver names + WhatsApp drafts (English + Arabic placeholder for Phase 9) +
       Approve/Edit/Dismiss buttons. Identical contract to /decisions DecisionCard but compact.
06:35  Owner edits one draft (clicks Edit → drawer opens), shortens it, hits Save & Approve.
       AgentAction row written: source=chat, threadId=…, msgId=…, toolName=draftCourierMessage
06:35  Owner clicks "Pin to Home" on the bar_chart view → toast "Pinned to Decisions header.".
       PinnedView row written: viewType=bar_chart, spec={query, params}, sortOrder=last+1.
06:36  Owner closes laptop. Total time on /chat: 4 minutes. Total time across morning ritual:
       ~7 minutes (down from 20 min Excel + WhatsApp baseline).
```

### 1.4 User Flow — Dispatcher's quick lookup via ⌘K (no /chat)

```
14:12  Dispatcher hits ⌘K on /floor (when /floor exists in Phase 7; in Phase 4, /v2/triage)
14:12  Types "where is driver Mohamed Khaled?" → Enter
```

The palette is **answer-first**, not chat-first. It only takes the user to `/chat` when the answer demands a richer view.

#### 3.1.3 Behaviour

| Interaction | Behavior |
|-------------|----------|
| Type query + Enter | Calls `POST /api/chat/threads` (creates thread), then `POST /api/chat/threads/{id}/messages` with the question; **auto-navigates to `/chat?thread={id}`** in the same tab (replaces history entry). The palette closes. The full-screen page shows the streaming response. |
| Click recent thread | Navigates to `/chat?thread={id}`; palette closes. |
| Click quick action | Same as type-and-Enter, but the question is a preset string (per-action below). |
| `>` prefix | Tags the message as **action-mode** for the agent — UI shows an `[action]` amber chip in the input row (already implemented today). Tells the agent to bias toward action proposals. |
| `?` prefix | Tags as **query-mode** — UI shows `[query]` sky chip. Bias toward read-only answers. |
| Click "View all in /chat" | Navigates to `/chat` (no thread param). |

#### 3.1.5 Loading State

When the user submits and the WS connection is establishing:
- Input is disabled, `Loader2` spin replaces send icon.
- Below the input, a 2-line skeleton appears for 600ms max — then the page navigates.

#### 3.1.6 Error State

If `POST /api/chat/threads/{id}/messages` returns 4xx/5xx:
- Toast (`Toast` provider) fires: variant `error`, body `"Couldn't reach Darb. {server message}"`.
- Input remains active for retry.
- Palette stays open.

#### 3.2.4 Generated View Blocks (`<GeneratedViewBlock />`)

The viewBlock is the heart of generative UI. Each block has a stable shape:

#### 3.2.8 Streaming State (`<StreamingIndicator />`)

When the user sends a message, the assistant message bubble first renders empty, then progresses through these states (driven by WebSocket events — see §8.3):

| State | Visual |
|-------|--------|
| `queued` | Pulsing dot animation in bubble: `· · ·` (6px sand-500 dots, 1.5s loop) |
| `tool_running` (per tool) | Tool-call chip in `running` state at the bottom of the bubble: `🔧 cashOutstanding running…` |
| `streaming_text` | Text appears character-by-character as it arrives over WS; cursor `|` blinks at end |
| `streaming_view` | viewBlock fades in at 60% opacity, fills 100% over 250ms |
| `complete` | All animation stops; meta footer (tokens, latency) populates |
| `error` | Red `AlertCircle` icon + "Couldn't generate answer. Retry?" with `[Retry]` button |

The composer stays disabled during `queued`/`tool_running`/`streaming_*`; user can hit `Esc` to cancel mid-stream (sends a `cancel` message over WS).

## 5. Interaction Patterns

### 5.1 Optimistic Updates

| Surface | Behavior |
|---------|----------|
| Send message | User bubble appears immediately. If server returns error → bubble flips to `failed` with retry. |
| Approve proposal in chat | Identical to /decisions: optimistic flip → 5s undo → AgentAction row written. On server error → revert + toast. |
| Pin to Home | Right panel shows `[Pin to Home]` button with spinner. On 200 → close panel + toast + the viewBlock's "Pin to Home" button changes to `[Pinned ✓]`. On error → toast + button stays active. |
| Schedule briefing | Form submit → spinner → on 200 → toast + close. On error → toast + form stays open. |
| Rename thread | Inline edit in header; save on blur or Enter. Optimistic; reverts on error. |
| Delete thread | Confirm modal → optimistic remove from sidebar → server call. On error → restore + toast. |

### 5.2 Real-time Updates (WebSocket — the Phase 4 deliverable)

WebSocket transport replaces the SSE polling that Phase 2 used for the Decisions inbox.

**Connection lifecycle:**
1. On `/chat` mount: open WS to `wss://{host}/api/chat/stream` with JWT in initial connect message.
2. Server validates JWT + tenantScope, returns `connected` event with `userId`, `tenantId`.
3. Client subscribes to a thread on demand: `{ type: "subscribe", threadId }`.
4. Server pushes events for that thread (see §8.3).
5. On disconnect: exponential backoff retry (1s, 2s, 4s, 8s, 16s, 30s cap).
6. After 3 failed reconnects: show fallback toast + switch to HTTP polling.

**SSE preserved:** existing `/api/notifications/stream` continues to work for notification toasts (out-of-band from chat).

### 5.3 Pin-to-Home Mechanic Detail

```
User clicks Pin to Home on a viewBlock
   ↓
Right panel slides in (250ms ease-sierra-out)
   ↓
PinPreviewPanel renders the tile at exact target dimensions
   ↓
User edits title (auto-populated), picks position, picks refresh frequency
   ↓
User clicks [Pin to Home]
   ↓
POST /api/pinned-views { tenantId, userId, viewType, spec, title, description, sortOrder, refreshFrequency }
   ↓
On 200:
  · Right panel closes
  · viewBlock's pin button changes to [Pinned ✓]
  · Toast: "Pinned to Decisions header. Open /decisions to view." [Open]
  · MetricEvent row: { event: "pin_to_home", viewType, threadId, msgId } — DEC-add-metric-events
   ↓
Next time user lands on /decisions:
  · GET /api/pinned-views returns the user's pinned views
  · PinnedViewsRail renders with the new tile
  · If refreshFrequency === "live", a 5-min interval refreshes the tile in place
```

### 5.4 Action Proposal Flow in Chat

```
User: "Apply a 10 KD penalty to drivers who missed shift today"
   ↓
Assistant streams response:
  1. Tool calls: liveFleetStatus → attendanceForPeriod → courierProfile×N
  2. table viewBlock: 4 drivers, each with proposed deduction
  3. ChatProposalCard rendered below the assistant bubble
       — toolName: applyPenalty (NOTE: applyPenalty is a Phase 8 tool, not Phase 4 scope)
       — proposalState: pending
   ↓
In Phase 4 reality, the only chat-callable action tool is draftCourierMessage. For applyPenalty
proposals in Phase 4, the ChatProposalCard renders with [Approve] disabled and a tooltip
"Action tool ships in Phase 8 — your approval is recorded for training only."
This mirrors Phase 2 §3.1.2 "Note for Phase 2 scope" semantics exactly.
   ↓
User clicks Approve (only enabled for draftCourierMessage in Phase 4)
   ↓
POST /api/decisions/{proposalId}/approve { source: "chat", threadId, msgId }
   ↓
AgentAction row written: source=chat, threadId, msgId
   ↓
ChatProposalCard transitions to "approved" state (5s undo window via toast)
```

### 5.5 Streaming Token Rendering

To avoid React reconciliation thrash on token streams, the assistant message's text content is rendered through an **incremental string state** (not a re-render per token):

- A `useState<string>` holds the streaming text.
- Each WS `text_delta` event appends to the string with `setStreamingText(prev => prev + delta)`.
- React batches updates in 16ms windows (browser default).
- At ~50 tokens/second this delivers smooth streaming with no janky re-renders.
- On `streaming_complete` event, the string is committed to the message record (via React Query mutation) and the streaming state clears.

### 5.6 Search Behaviour

Thread search calls `GET /api/chat/threads?search={q}` with debounced input (300ms). The query searches:
- Thread `title` (full-text)
- Thread `preview` (full-text)
- All `ChatMessage.content` within the user's threads (full-text via Postgres `tsvector` index — backend ships the index)

Result rendering: replaces the standard "Today / Yesterday / This week" grouping with a flat list ranked by relevance. Highlights matched terms in `<mark className="bg-amber-200">`.

### 5.7 Conflicting Proposals

If a user approves the same proposal in `/decisions` and `/chat` simultaneously (e.g., multi-tab), the Phase 2 optimistic-lock returns 409 on the second approve. UI behavior: chat ChatProposalCard shows "Already approved by you {time} ago — see audit log" with link to `/decisions/audit`.

### 5.8 Cross-tab Sync

When the user has `/decisions` open in tab A and `/chat` open in tab B:
- A new chat-approved AgentAction in tab B fires a `BroadcastChannel('darb.audit')` message.
- Tab A's audit log polls or refreshes the audit count badge.
- Decisions tab does NOT auto-refresh the inbox (would disrupt user reading); a small "1 new action" toast appears with [Refresh] link.

Connection handshake:

Subscribe to a thread:
```
Client → Server:

Server → Client:
  { type: "subscribed", threadId }
```

Streaming events (server → client):
```typescript
{ type: "queued", msgId, threadId }
{ type: "tool_start", msgId, toolCallId, toolName, input }
{ type: "tool_complete", msgId, toolCallId, toolName, output, latencyMs }
{ type: "tool_error", msgId, toolCallId, toolName, error }

{ type: "view_block", msgId, view: GeneratedView }
{ type: "proposal", msgId, proposal: ChatProposalRef }
{ type: "complete", msgId, meta: { promptTokens, completionTokens, latencyMs, modelName } }
{ type: "error", msgId, error: string }
{ type: "cancelled", msgId }
```

Cancel mid-stream:
```
Client → Server:
  { type: "cancel", msgId }
```

Heartbeat:
```
Both sides exchange { type: "ping" } every 30s; missed → reconnect.
```

## 9. Edge Cases

| Case | Behavior |
|------|----------|
| **WS connection refused** (server down) | Persistent toast + auto-retry with backoff. Composer disabled. After 3 failures, switch to HTTP polling fallback. |
| **WS connection drops mid-stream** | Existing assistant message bubble shows "Reconnecting…" indicator + spinner. On reconnect, resume from last `text_delta` cursor (server tracks cursors per msgId). |
| **Cross-tab WS conflict** (same user, two tabs subscribed to same thread) | Both tabs receive identical events. UI is idempotent (each event has a sequence number; duplicates ignored). |
| **Thread not found** (deleted or wrong tenant) | `<ErrorState />` rendered in thread pane: "Thread not found. It may have been deleted." with link back to `/chat`. |
| **User opens /chat with VIEWER role** | Read-only: thread sidebar shows own threads + briefings shared to user; composer is replaced with a banner "Read-only access — contact admin to send messages." |
| **Thread title rename to empty string** | Defaults back to auto-generated title from first user message. |
| **viewBlock spec is malformed** (server bug) | Renders a placeholder `<CalloutView tone="warning">` saying "Couldn't render this view (technical detail). Refresh to retry." Hidden detail in dev console. |
| **Pin-to-Home with same title as existing pin** | Allow duplicate (pinned views are user-scoped; no uniqueness constraint on title). Toast: "Pinned (you have 2 views with this title)." |
| **Pin-to-Home over the implicit limit** (e.g., 50+ pinned views) | Soft cap of 24 per user; on attempt: toast warning "You have 24 pinned views. Consider unpinning some."; allow override via "Pin anyway". |
| **Schedule briefing with invalid cron** | Validation error in form (red text below cron input): "Cron expression is not allowed in v1. Choose 06:00 or 07:00, or contact admin for custom schedules." |
| **Schedule briefing fires at wrong time** (timezone bug) | Server stores cron in tenant timezone (Asia/Kuwait UTC+3). Worker uses tenant timezone. Defensive: brief shows the "Generated at {timeInTenantTz}" footer. |
| **Approving a chat proposal whose card was already dismissed in /decisions** | Backend optimistic-lock returns 409. UI: ChatProposalCard shows "Already dismissed in /decisions {time}" with link to audit. Tag turns sand. |
| **Sending a message when previous assistant is still streaming** | Composer is disabled until `complete`/`error`/`cancelled`. User can hit `Esc` to cancel mid-stream. |
| **User clicks Cancel mid-stream** | WS sends `cancel` event. Server stops Anthropic stream. Assistant message marked `state: "cancelled"`. UI: bubble dimmed with "Cancelled" pill + retry button. |
| **Tool call hangs >30s** | Server times out, fires `tool_error` event. UI: red ✗ chip + assistant message includes a callout "I had trouble reaching {tool}. Here's what I have anyway: …". |
| **Anthropic API quota exceeded** | Server returns clean 429. UI toast: "Darb is rate-limited. Try again in a minute." Composer disabled for 60s with countdown. |
| **Long prompt (>16k tokens)** | Server returns 413. UI toast: "That message is too long. Try splitting it up." |
| **Long thread (>200 messages)** | Sidebar shows full thread; thread pane only renders last 50 by default with "[Load older messages]" button. |
| **Empty assistant response** (model returns nothing) | Render single callout (info): "I didn't have anything to add. Try rephrasing?" |
| **Mobile device** (<768px width) | `/chat` renders an empty state: "Chat is best on desktop. Open Darb on a laptop for the full experience. Tap below to view your Decisions inbox." with link to `/decisions`. |
| **User has 0 threads, 0 pinned views** (cold start) | `/chat` empty state with quick actions; `/decisions` Pinned-views rail is hidden entirely. |
| **Briefing fires when user is offline** | Thread is created server-side with the assistant's response written. On next user load, a "Briefings (1)" group at top of sidebar shows the unread brief with a primary dot. |
| **Conflicting pin** (user pins same view twice from same message) | Backend: deduplicates by `(userId, sourceMessageId, viewBlockId)`. Returns existing PinnedView with toast "Already pinned." |
| **Right panel open + user clicks outside** | If `<ProposalEditDrawerInline />` has unsaved changes → `<ConfirmModal />`: "Discard your edits?" If clean → close immediately. |
